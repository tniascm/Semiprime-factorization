# Pillar 4: Linear Algebra – Block Wiedemann & Tensor Core GF(2) Matrix Multiplication

This document details the fourth pillar of the `rust-nfs` architecture, which focuses on revolutionizing the Linear Algebra (LA) stage of the Number Field Sieve (NFS) algorithm.

## Overview

The Linear Algebra stage of NFS involves finding the nullspace of a massive, sparse, binary matrix (over GF(2)). Historically, implementations like CADO-NFS have relied on Gaussian Elimination (GE) for smaller matrices and the Block Wiedemann algorithm for larger ones, typically executing on CPU clusters.

However, as bit sizes scale towards 128 bits and beyond, GE becomes unviable due to its $O(n^3)$ complexity. While Block Wiedemann is $O(n^2)$ and suited for sparse matrices, executing it on CPUs leaves tremendous performance potential on the table.

Our "Unequivocal Domination" strategy shifts this paradigm entirely by leveraging modern GPU architectures, specifically utilizing Tensor Cores to perform GF(2) arithmetic at unprecedented speeds.

## The Block Wiedemann Algorithm

The Block Wiedemann algorithm is an iterative method for finding vectors in the kernel of a sparse matrix $M$ over a finite field (in our case, GF(2)). Instead of performing row operations, it repeatedly multiplies the sparse matrix by blocks of dense vectors.

### Key Phases:
1.  **Sequence Generation (Krylov Subspace):** Repeatedly compute $M^i \cdot V$ where $V$ is a block of random vectors. This is the most computationally intensive part, consisting almost entirely of sparse matrix-vector multiplications (SpMV).
2.  **Berlekamp-Massey / Matrix Polynomial:** Find linear recurrence relations in the generated sequence to construct a matrix polynomial.
3.  **Evaluation:** Compute the final kernel vectors using the polynomial coefficients and the matrix $M$.

## Tensor Core GF(2) Matrix Multiplication

The true leap in performance comes from how we execute the sequence generation phase. Modern GPUs (like NVIDIA's Ampere, Hopper, etc.) feature Tensor Cores designed for rapid, low-precision matrix multiplication (e.g., FP16, INT8).

### The 1000x Shift Mechanism

While Tensor Cores natively operate on floating-point or integer arithmetic, we can cleverly encode GF(2) (binary) arithmetic into these operations to achieve Tera-Operations Per Second (TOPS) that no CPU cluster can match.

1.  **Encoding GF(2) into INT8:** We can pack multiple GF(2) bits into the standard INT8 format expected by the Tensor Cores.
2.  **Exploiting Hardware:** By structuring our Block Wiedemann dense vector blocks to align with the dimensions expected by Tensor Core instructions (e.g., mma.sync), we can perform massive amounts of GF(2) dot products simultaneously.
3.  **Avoiding the Bottleneck:** This approach guarantees that the Linear Algebra stage never becomes the bottleneck as the semiprime size $N$ scales up. What used to take days on a CPU cluster can be reduced to hours or minutes on a single modern GPU.

## Implementation Guidelines for `rust-nfs`

To realize this vision in `rust-nfs`, the implementation must adhere to the following principles:

*   **Modular Design:** The Block Wiedemann logic must be isolated in a dedicated crate (e.g., `block-wiedemann`) to allow independent testing and benchmarking.
*   **Bit-Packed Representation:** The matrix and vector blocks should be aggressively bit-packed (e.g., using arrays of `u64`) to maximize memory bandwidth utilization before even reaching the GPU.
*   **Extensible to CUDA/Metal:** The initial CPU-based Rust implementation must be structured so that the core SpMV operations can be easily swapped out for FFI calls to optimized CUDA or Apple Metal kernels in the future.
*   **Zero-Copy Integration:** The matrix generated by the filtering stage should be passed to the LA stage without disk serialization, adhering to the overall zero-copy architecture of `rust-nfs`.

## Conclusion

By adopting the Block Wiedemann algorithm and eventually accelerating it with GPU Tensor Cores, `rust-nfs` will secure an insurmountable asymptotic advantage in the Linear Algebra phase, turning a traditionally painful bottleneck into a fast, highly parallelized operation.
